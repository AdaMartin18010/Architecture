## 1. 原始 COCOMO II 复用公式

```
开发工作量 (Person-Months):
    PM = A × (Size)^E × ∏(EMᵢ)

其中:
    A = 2.94 (校准常数，基于 1990-2000 年 161 个项目数据)
    Size = 千等效源代码行数 (KSLOC)
    E = B + 0.01 × Σ(SFⱼ)
    B = 0.91 (默认)
    SFⱼ = 5 个规模因子
    EMᵢ = 17 个工作量乘数

复用调整后的规模:
    ESLOC = ASLOC × (AAF) / 100

    AAF = 0.4 × (DM) + 0.3 × (CM) + 0.3 × (IM)

    DM = 设计修改百分比 [0-100%]
    CM = 代码修改百分比 [0-100%]
    IM = 集成工作量百分比 [0-100%]

若 AAF ≤ 50:
    ESLOC = ASLOC × [AA + AAF × (1 + 0.02 × SU × UNFM)] / 100

若 AAF > 50:
    ESLOC = ASLOC × [AA + AAF + (SU × UNFM)] / 100

    AA = 评估与同化百分比 (Assessment and Assimilation)
    SU = 软件理解增量 [10-50%]
    UNFM =  unfamiliarity 因子 [0.0-1.0]
```

## 2. 2026 校准动因

COCOMO II 的校准常数 A=2.94 基于 1990-2000 年的项目数据，对 2026 年的以下变化存在偏差：

| 变化 | 对 COCOMO II 的影响 | 校准方向 |
|------|-------------------|---------|
| AI 辅助编程 (Copilot/Cursor) | 编码速度提升 30-55%，但审查成本增加 | 降低 CM 权重，增加审查因子 |
| Serverless/FaaS | 基础设施代码大幅减少，但配置代码增加 | 重新定义 "代码行" 范围 |
| 低代码/无代码平台 | 可视化开发替代部分手写代码 | SLOC 不再适用，引入"功能点"换算 |
| 组件生态成熟 (npm/crates.io) | 依赖深度增加，但直接编写代码减少 | 调整 RUSE 乘数范围 |
| DevSecOps 流水线 | 自动化测试/扫描降低集成成本 | 降低 IM 基准值 |
| 远程协作/全球化 | 沟通成本变化 | 调整 TEAM 乘数 |

## 3. 2026 校准参数

### 3.1 AI 辅助开发调整

**AI 辅助编码影响**: 当项目使用 AI 辅助编码工具时，原始 CM 值应乘以 AI 效率因子：

```
CM_2026 = CM_original × AI_efficiency

AI_efficiency 取值:
├── 无 AI 辅助: 1.00
├── 基础代码补全 (GitHub Copilot): 0.75
├── 高级 Agent (Cursor, Claude Code): 0.60
├── 全栈 AI 生成 (vibe coding): 0.45
└── 但必须增加 AI_review_cost 因子到 IM:
    IM_2026 = IM_original × (1 + AI_review_cost)
    AI_review_cost ∈ [0.10, 0.30]（取决于安全关键程度）
```

### 3.2 Serverless 调整

Serverless 项目中，将"代码行"扩展为"等效功能单元 (EFU)":

```
Size_serverless = (SLOC_traditional × 0.3) + (SLOC_function_handlers) + (Config_lines × 0.1)

理由:
├── 传统基础设施代码减少 70%（托管服务替代）
├── Function handler 代码是传统代码的等效度量
└── 配置代码（YAML/Terraform）复杂度约为传统代码的 10%
```

### 3.3 低代码平台调整

低代码项目使用功能点换算：

```
Size_lowcode = FP × conversion_factor

conversion_factor:
├── 简单 CRUD 应用: 25 SLOC/FP
├── 中等复杂度业务应用: 40 SLOC/FP
├── 复杂集成应用: 60 SLOC/FP
```

### 3.4 复用成熟度 RUSE 调整

原始 RUSE 范围 0.65-1.00，2026 建议扩展至考虑 AI 和平台因素：

| RUSE 等级 | 原始值 | 2026 建议值 | 场景 |
|-----------|--------|------------|------|
| 极低 | 1.00 | 1.00 | 无复用，全部自研 |
| 很低 | 0.95 | 0.93 | 偶发复用，个人级 |
| 低 | 0.89 | 0.86 | 项目级复用 |
| 一般 | 0.84 | 0.80 | 组织级复用 + IDP |
| 高 | 0.78 | 0.73 | 产品线级复用 + AI 辅助 |
| 很高 | 0.72 | 0.66 | 跨组织复用 + SLSA L3+ |
| 极高 | 0.65 | 0.58 | 黑盒复用 + AI 自动适配 |

### 3.5 2026 综合校准公式

```
PM_2026 = A_2026 × (Size_2026)^E × ∏(EMᵢ_2026)

其中:
    A_2026 = 2.20 （建议下调，反映工具和生态成熟度）
    Size_2026 = KSLOC_2026（根据项目类型换算）
    EMᵢ_2026 包含:
        ├── RUSE_2026: 复用成熟度
        ├── AI_tool: AI 辅助开发因子 [0.85-1.00]
        ├── AI_review: AI 审查成本因子 [1.10-1.30]
        └── 其他 14 个传统乘数
```

## 4. 完整计算示例

### 场景：电商订单系统的复用成本估算（2026 版）

**项目参数**:

- 新开发订单系统预计需要 10,000 SLOC
- 组织内已有库存管理系统的订单处理模块，共 3,000 SLOC
- 计划复用该模块，需进行以下改编:
  - 设计修改: 30%（适配新的数据库 Schema）
  - 代码修改: 15%（AI 辅助下从 20% 降低）
  - 集成工作量: 50%（新的消息队列集成）
- AI_review_cost = 0.20
- AA = 10%（评估与同化）
- SU = 20%（软件理解）
- UNFM = 0.5
- 组织复用成熟度: 高（产品线级复用 + AI 辅助）

**计算**:

```
1. 改编调整因子:
   AAF = 0.4 × 30% + 0.3 × 15% + 0.3 × 50%
       = 0.12 + 0.045 + 0.15 = 0.315

2. AI 调整后:
   CM_effective = 15% (已经包含 AI 效应)
   IM_effective = 50% × (1 + 0.20) = 60%

   AAF_2026 = 0.4 × 30% + 0.3 × 15% + 0.3 × 60%
            = 0.12 + 0.045 + 0.18 = 0.345

3. 非线性调整 (AAF ≤ 0.5):
   ESLOC = 3,000 × [10 + 34.5 × (1 + 0.02 × 20 × 0.5)] / 100
         = 3,000 × [10 + 34.5 × 1.2] / 100
         = 3,000 × 51.4 / 100
         = 1,542 SLOC

4. 总项目规模:
   Size = (10,000 - 3,000) + 1,542 = 8,542 SLOC = 8.542 KSLOC

5. 规模因子评分:
   PREC = 4.0, FLEX = 3.0, ARCH = 3.0, TEAM = 3.0, PMAT = 3.0
   Σ(SFⱼ) = 16.0
   E = 0.91 + 0.01 × 16.0 = 1.07

6. 工作量乘数（仅展示关键项）:
   RUSE_2026 = 0.73 (高复用)
   AI_tool = 0.85 (使用高级 Agent)
   AI_review = 1.20
   ∏(EMᵢ) ≈ 0.73 × 0.85 × 1.20 ≈ 0.745

7. 开发工作量:
   PM = 2.20 × (8.542)^1.07 × 0.745
      = 2.20 × 9.78 × 0.745
      ≈ 16.0 人月

对比（无复用，无 AI）:
   PM_no_reuse = 2.94 × (10.0)^1.07 × 1.0
               = 2.94 × 11.75 × 1.0
               ≈ 34.5 人月

对比（有复用，无 AI）:
   PM_reuse_no_ai = 2.94 × (7.99)^1.07 × 0.78
                  ≈ 20.9 人月

2026 版节约:
   相对无复用: 34.5 - 16.0 = 18.5 人月 (53.6%)
   相对传统复用: 20.9 - 16.0 = 4.9 人月 (23.4%)
   → AI 辅助额外节约 23.4%
```

## 5. RUSE 乘数 2026 建议值

| RUSE 等级 | 描述 | 2026 建议值 | 复用场景 |
|-----------|------|------------|----------|
| **极低** | 无复用，全部自研 | 1.00 | 创新项目、无先例 |
| **很低** | 偶发复用，无系统管理 | 0.93 | 个人项目、临时复用 |
| **低** | 项目级复用，有管理 | 0.86 | 团队内共享代码 |
| **一般** | 组织级复用，标准化 + IDP | 0.80 | 企业级组件库 |
| **高** | 产品线级复用 + AI 辅助 | 0.73 | 产品线工程 |
| **很高** | 跨组织复用 + SLSA L3+ | 0.66 | 行业级共享组件 |
| **极高** | 黑盒复用 + AI 自动适配 | 0.58 | COTS/SaaS + AI 集成 |

## 6. 局限性声明

> **公理 V.1** (Value Quantification Uncertainty): 复用的价值量化存在固有的不确定性。COCOMO II 的校准常数基于历史数据，对 AI 辅助开发、Serverless 架构、低代码平台的适用性存在偏差。任何量化结果应被视为"最佳估计"而非"精确真理"。

> **公理 V.2** (Strategic Value Non-Quantifiability): 复用的战略价值（上市时间优势、生态系统网络效应、组织能力进化）难以用货币精确量化，但不可因此忽略。

> **定理 V.1** (ROI Threshold): 复用项目的 ROI 为正的必要条件是：复用资产的改编调整因子 AAF < 0.7。若 AAF ≥ 0.7，复用的直接经济价值消失，仅剩战略价值。
