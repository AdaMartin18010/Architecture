# 应用服务复用模式

> **版本**: 2026-06-12
> **定位**: 03-application-architecture-reuse / 03-app-service
> **对齐标准**: SOA, OASIS SOA Reference Architecture, NIST SP 800-204

---

## 核心概念

应用服务复用关注**系统级服务**的复用：将业务能力封装为可共享、可编排、可治理的应用服务。它介于业务服务（业务语义）与组件/功能（技术实现）之间，强调通过标准化契约实现跨应用集成。

| 概念 | 定义 | 与微服务的区别 |
|:---|:---|:---|
| **应用服务（Application Service）** | 为应用系统提供可复用功能的服务单元，通常对应一个业务能力子集 | 不一定要求独立部署，可存在于应用内部 |
| **服务契约（Service Contract）** | 服务的接口、数据、质量、安全约定 | 与微服务契约等价，但治理范围更广 |
| **服务编排（Service Orchestration）** | 按流程组合多个服务完成业务目标 | 可跨单体应用和微服务 |
| **服务目录（Service Catalog）** | 组织级应用服务的统一注册与发现机制 | 是复用治理的基础设施 |

---

## 复用模式

### 模式 1：服务目录驱动复用

建立组织级应用服务目录，包含：

- 服务名称、版本、所有者、SLA
- 接口规范（OpenAPI/AsyncAPI/WSDL）
- 依赖关系与消费方清单
- 合规状态（安全扫描、许可证检查）

### 模式 2：共享 API 网关

通过 API 网关暴露可复用服务：

- 统一认证、限流、缓存
- 协议转换（REST/gRPC/SOAP）
- 灰度发布与版本路由

### 模式 3：服务门面（Service Facade）

为复杂遗留系统提供简化门面：

- 隐藏内部复杂性
- 提供稳定契约
- 支持新旧系统共存

### 模式 4：组合服务（Composite Service）

将多个细粒度服务组合为粗粒度可复用服务：

- 降低消费方调用复杂度
- 聚合跨域数据
- 通过缓存提升性能

---

## 与微服务复用的关系

| 维度 | 应用服务复用 | 微服务复用 |
|:---|:---|:---|
| 粒度 | 较粗，可对应一个应用或子系统 | 较细，独立部署单元 |
| 部署 | 可内嵌于应用 | 独立部署 |
| 治理重点 | 服务目录、契约、编排 | 边界、自治、版本、服务网格 |
| 适用场景 | 企业应用集成、SOA、遗留系统 | 云原生、DevOps、快速迭代 |
