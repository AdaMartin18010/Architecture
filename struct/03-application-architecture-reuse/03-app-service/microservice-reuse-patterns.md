# 微服务架构中的复用模式

> **版本**: 2026-06-06
> **定位**: 系统梳理微服务架构下的复用模式、反模式与决策框架

---

## 1. 微服务复用的核心张力

微服务架构天然强调"服务边界内的自治"，而复用本质上要求"跨边界的共享"。二者之间存在三个核心张力：

| 张力 | 描述 | 平衡策略 |
|------|------|---------|
| **自治 vs 共享** | 服务自治要求独立部署、独立演进；复用要求共享代码、共享数据契约 | 通过库共享稳定契约，通过服务调用共享不稳定行为 |
| **内聚 vs 解耦** | 高内聚要求相关功能聚合；解耦要求功能分散 | 按业务能力划分服务 |
| **演进速度 vs 兼容性** | 快速演进要求频繁变更；复用要求向后兼容 | SemVer + 契约测试 |

> **定理 3.1** (Microservice Decomposition Limit): 微服务拆分的收益边际递减。存在一个最优服务数量 N*，使得继续拆分带来的复用收益被协调成本抵消。

## 2. 四种复用模式

### 模式 A: Shared Library（共享库）

**适用场景**:

- 纯工具函数
- 稳定的数据结构定义
- 客户端 SDK

**风险**: 版本冲突、共享库膨胀

### 模式 B: Shared Service（共享服务）

**适用场景**:

- 用户认证/授权
- 通知服务
- 支付服务

**风险**: 性能瓶颈、变更协调

### 模式 C: Sidecar / Proxy Pattern

**适用场景**:

- 横切关注点（日志、监控、安全）
- 跨语言服务统一治理

**代表技术**: Istio, Linkerd, Dapr

### 模式 D: Platform Capability（平台能力）

**适用场景**:

- 新服务脚手架
- CI/CD 流水线模板
- 可观测性配置模板

**代表技术**: Backstage, Terraform, Helm

---

## 3. 模式选择决策树

```
复用模式选择
│
├── 复用内容是代码还是运行时行为？
│   ├── 代码 → Shared Library
│   └── 运行时 → Shared Service / Sidecar / Platform
│       ├── 横切关注点 → Sidecar
│       ├── 频繁变更 → Shared Service
│       └── 标准化模板 → Platform Capability
```

---

## 4. 关键反模式

| 反模式 | 症状 | 重构策略 |
|--------|------|---------|
| Distributed Monolith | 多个服务必须同时部署 | 识别真正边界，引入事件驱动 |
| Shared Database | 多个服务访问同一数据库 | 每个服务独立数据存储 |
| Over-Engineered Reuse | 复用成本高于自研 | 重新评估复用收益 |
| Version Hell | 多版本冲突 | 版本锁定、私有仓库 |
