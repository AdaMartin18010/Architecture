| 维度 | 内容 |
|------|------|
| **时间** | 2024-03 |
| **攻击者** | Jia Tan（伪装成合法贡献者，潜伏 2021-2024） |
| **攻击技术** | 源码混淆（宏+条件编译）、测试文件投毒、构建时注入、IFUNC 解析器劫持 |

## 防御策略矩阵

### 预防 (Prevention)

| 策略 | 实施要点 | 对应 SLSA 等级 |
|------|---------|---------------|
| 依赖最小化 | 仅引入必要依赖，定期清理 | L1+ |
| 依赖锁定 | 使用 lockfile 确保可复现构建 | L1+ |
| 私有仓库 | 优先使用内部私有仓库 | L1+ |
| 命名空间保护 | 注册组织命名空间 | L1+ |
| 代码签名 | GPG 提交签名、cosign 构建产物签名 | L2+ |
| 构建隔离 | 托管 CI/CD，隔离外部网络 | L3 |
| 可复现构建 | 相同输入 → 相同输出 | L4 |

### 检测 (Detection)

| 策略 | 工具/方法 | 频率 |
|------|----------|------|
| SBOM 生成 | Syft, Trivy, GitHub Dependency Graph | 每次构建 |
| 漏洞扫描 | Snyk, OWASP Dependency-Check, Dependabot | 持续监控 |
| 静态分析 | Semgrep, CodeQL, SonarQube | 每次提交 |
| 行为监控 | Falco, runtime anomaly detection | 运行时 |
| 性能监控 | 性能回归测试 | 每次发布 |

### 响应 (Response)

| 策略 | 目标 | 时间要求 |
|------|------|---------|
| 快速隔离 | 阻断受影响组件进入生产 | < 1 小时 |
| 影响分析 | 通过 SBOM 定位受影响系统 | < 4 小时 |
| 补丁管理 | 自动化补丁测试、灰度发布 | < 24 小时 |
| 事件取证 | 保留构建日志、SBOM、网络流量 | 立即 |
| 通报机制 | 内部通报、CVE 发布、客户通知 | < 72 小时 |

## 关键定理

> **公理 S.1** (Trust Transitivity Collapse): 软件供应链中的信任是传递的，但传递链的长度与信任度成指数反比。形式化：Trust(A, M) = ∏ Trust(Xᵢ, Xᵢ₊₁) ≈ 0 当链长度 > 5。

> **公理 S.2** (SBOM Completeness Illusion): SBOM 的完整性是一种幻觉。SBOM 只能记录已知的依赖，无法记录隐式依赖（编译器运行时、操作系统库、硬件微码）。完全的供应链透明是不可达到的。

> **定理 S.1** (SLSA Level Monotonicity): 若组件 A 依赖组件 B，则 SLSA(A) ≤ SLSA(B)。任何试图提升 A 的 SLSA 等级而不提升其依赖的尝试是徒劳的。

> **定理 S.2** (XZ Backdoor Detection): 供应链后门的检测概率与以下因素正相关：构建可复现性、性能监控粒度、代码审查深度、贡献者验证强度；与以下因素负相关：代码复杂度、构建系统黑箱度、社区压力。
