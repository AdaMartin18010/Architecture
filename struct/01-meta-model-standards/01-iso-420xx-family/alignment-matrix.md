## 1. 标准族谱

```text
ISO/IEC/IEEE 42010:2022  ──→ 架构描述 (Architecture Description)
         │                    └── 视图(View)、视点(Viewpoint)、利益相关者(Stakeholder)
         │                    └── 架构描述框架 (ADF) = 视点库 + 视图库 + 模型库 + 架构理由
         │
         ├── ISO/IEC/IEEE 42020:2019 ──→ 架构过程 (Architecture Processes)
         │   ├── 架构开发过程 (ADP)
         │   ├── 架构管理过程 (AMP)
         │   └── 架构治理过程 (AGP)
         │
         ├── ISO/IEC/IEEE 42030:2019 ──→ 架构评估 (Architecture Evaluation)
         │   ├── 评估框架 (Evaluation Framework)
         │   ├── 评估准则 (Criteria)
         │   └── 评估方法 (Methods)
         │
         ├── ISO/IEC/IEEE DIS 42024 ──→ 架构基础 (Architecture Fundamentals)
         │   └── 概念、术语、原则的元定义
         │
         └── ISO/IEC/IEEE DIS 42042 ──→ 参考架构 (Reference Architectures)
             └── 参考架构的构建、描述、使用与演化规范
```

## 2. 核心概念映射

| 元概念 | ISO/IEC/IEEE 42010:2022 定义 | 复用视角映射 | 形式化约束 |
|--------|------------------------------|--------------|------------|
| **架构 (Architecture)** | 系统在其环境中的基本概念或属性，体现为它的元素、关系以及设计和演进的原则 | 复用不是附属特性，而是架构的**结构性约束** | Architecture = ⟨元素, 关系, 原则, 环境⟩ |
| **架构描述 (AD)** | 表达架构的工作产物 | 复用契约的载体（规格说明、接口定义、variability 模型） | AD = ∪视图ᵢ(视点ᵢ) |
| **视点 (Viewpoint)** | 针对利益相关者关注点的架构描述约定 | 业务视点、应用视点、组件视点、功能视点 | 视点 = ⟨利益相关者, 关注点, 语言, 方法⟩ |
| **视图 (View)** | 从特定视点生成的架构描述 | 业务架构视图、应用架构视图、组件架构视图、功能架构视图 | 视图 = 视点(模型) |
| **利益相关者 (Stakeholder)** | 对系统有个人、团队或组织利益的人 | 业务分析师、应用架构师、组件工程师、功能开发者 | 利益相关者 = {业务, 应用, 组件, 功能} × {所有者, 使用者, 评估者} |
| **关注点 (Concern)** | 利益相关者对系统的利益 | 业务一致性、应用可替换性、组件可组合性、功能可复用性 | 关注点 ⊆ 质量属性 × 业务目标 |
| **模型 (Model)** | 系统某方面的简化表示 | 复用的核心语义单元 | 模型 = ⟨语法, 语义, 语用⟩ |
| **架构理由 (Rationale)** | 架构决策的理由 | 复用决策的追溯依据 | 理由 = {问题, 方案, 决策, 影响} |

## 3. 42020 架构过程与复用的集成

| 42020 过程 | 复用活动 | 输入 | 输出 | 负责角色 |
|-----------|---------|------|------|---------|
| **架构开发 (ADP)** | 识别复用机会、选择复用资产 | 需求规格、现有资产目录 | 复用决策、适配设计 | 架构师 |
| **架构管理 (AMP)** | 维护资产库、版本控制、兼容性矩阵 | 资产变更请求 | 更新后的资产目录 | 资产管理员 |
| **架构治理 (AGP)** | 复用准入评估、合规检查、质量审计 | 候选资产、组织策略 | 评估报告、复用授权 | 治理委员会 |

## 4. 42030 架构评估在复用中的应用

```text
42030 评估框架在复用中的应用
├── 1. 评估上下文定义
│   ├── 评估目标: 确定候选资产是否满足目标上下文需求
│   ├── 评估范围: 语义/技术/质量/安全/许可/成本
│   └── 利益相关者: 资产提供方、使用方、评估者
│
├── 2. 评估准则定义
│   ├── 功能适合性 (ISO 25010)
│   ├── 可复用性 (Reusability)
│   ├── 可维护性 (Maintainability)
│   ├── 安全性 (Security)
│   ├── 兼容性 (Compatibility)
│   └── 可移植性 (Portability)
│
├── 3. 评估方法选择
│   ├── SAAM (Software Architecture Analysis Method)
│   ├── ATAM (Architecture Tradeoff Analysis Method)
│   ├── CBAM (Cost Benefit Analysis Method)
│   └── 自定义复用评估矩阵
│
└── 4. 评估证据收集
    ├── 测试覆盖率报告
    ├── SBOM / SLSA Provenance
    ├── 架构描述文档
    ├── 许可证合规声明
    └── 复用历史记录
```

## 5. DIS 42024 / DIS 42042 的前瞻对齐

| 草案标准 | 核心内容 | 对复用体系的影响 | 当前状态 |
|----------|---------|-----------------|---------|
| **DIS 42024** | 架构基础：概念、术语、原则的元定义 | 为元模型层提供更严格的形式化基础 | 草案阶段，预计 2026-2027 发布 |
| **DIS 42042** | 参考架构：构建、描述、使用与演化 | 为可复用参考架构提供标准化模板 | 草案阶段，预计 2026-2027 发布 |

**关键影响**:

- 42024 可能将"参考架构"明确纳入元模型，强化 `03-application-architecture-reuse` 的理论基础
- 42042 将为"产品线参考架构"提供 ISO 级别的规范，直接对齐 `01-meta-model-standards/03-iso-26550-ple`

## 6. 与 ISO 26550:2015 的交叉映射

| ISO 42010 概念 | ISO 26550 对应概念 | 复用视角解释 |
|---------------|-------------------|-------------|
| Architecture Description | Domain Engineering Output | 领域工程的产物是可复用的架构描述 |
| Viewpoint | Variability Viewpoint | 变性视点专门描述共性与变性的分离 |
| Model | Core Asset | 模型是核心资产的主要表现形式 |
| Concern | Stakeholder Requirement | 利益相关者关注点驱动复用需求 |
| Rationale | Variability Binding Decision | 架构理由记录变性绑定的决策依据 |
| ADM / Process | Domain Eng. + Application Eng. | 42020 过程可映射到产品线工程双轨 |

## 7. 关键公理-定理（元模型层）

> **公理 M.1** (Architecture-Reuse Duality): 架构的本质是**约束的集合**；复用的本质是**约束的传递**。一个架构的可复用性等于其约束的**可传递性**与**可组合性**的乘积。
> **公理 M.2** (Variability Axiom): 复用的本质是管理**共性 (Commonality)** 与**变性 (Variability)** 的分离与绑定。没有变性管理的复用是克隆，不是工程。
> **公理 M.3** (Hierarchy Non-Reduction): 复用具有层次性（业务→应用→组件→功能），层次间**不可约化**。业务复用不能降维为组件复用，功能复用不能升维为应用复用。
> **定理 M.1** (Viewpoint Composition): 若视点 V₁ 覆盖关注点 C₁，V₂ 覆盖 C₂，且 C₁ ∩ C₂ ≠ ∅，则复合视点 V₁₂ = V₁ ⊗ V₂ 必须显式定义 C₁ ∩ C₂ 的**冲突消解机制**。
> **定理 M.2** (Standard Alignment Transitivity): 若标准 S₁ 对齐 S₂，S₂ 对齐 S₃，则 S₁ 与 S₃ 的对齐关系需通过 S₂ 的**概念映射表**显式定义，不可默认传递。
