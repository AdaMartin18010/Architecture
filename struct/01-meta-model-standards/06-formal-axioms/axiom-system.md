# 形式化公理体系

> **版本**: 2026-06-06
> **定位**: 建立"软件工程架构复用视角"的元层公理系统，为全体系提供逻辑基础

---

## 1. 元公理 (Meta-Axioms)

### M.1 Architecture-Reuse Duality

> 架构的本质是**约束的集合**；复用的本质是**约束的传递**。

形式化：

```
设 A 为一个架构，A = ⟨C, R, V⟩
    C: 组件集合
    R: 组件间关系集合
    V: 作用于 C 和 R 的约束集合

则复用 Reuse(A, Ctx) 等价于：
    ∃ V' ⊆ V: V' 在上下文 Ctx 中仍然成立
```

**解释**: 当复用一个架构时，真正传递的不是组件本身，而是约束。组件可以替换，但约束必须保持。

### M.2 Variability Axiom

> 复用的本质是管理**共性 (Commonality)** 与**变性 (Variability)** 的分离与绑定。没有变性管理的复用是克隆，不是工程。

形式化：

```
设 S 为软件资产族，S = ⟨B, V, Γ⟩
    B: 不变的核心（Base）
    V: 可变的点集（Variation Points）
    Γ: 绑定规则，将 V 映射到具体变体

Reuse(S) = B ∪ Γ(V ↦ vᵢ)
```

**解释**: 可复用资产必须明确区分"什么是不变的"和"什么是可以变的"。

### M.3 Hierarchy Non-Reduction

> 复用具有层次性（业务→应用→组件→功能），层次间**不可约化**。某一层的复用失败不能通过另一层的优化完全弥补。

形式化：

```
设 L = {L_business, L_application, L_component, L_function}
∀ Lᵢ, Lⱼ ∈ L, i ≠ j:
    Reuse(Lᵢ) ⊄ Reuse(Lⱼ)
```

**解释**: 业务层的价值流复用不能替代组件层的接口设计；组件层的完美复用也不能弥补业务层的价值定义错误。

## 2. 存在性公理 (Existence Axioms)

### E.1 Reuse Asset Existence

> 并非所有软件资产都适合复用。可复用资产必须满足三个必要条件：
>
> 1. **稳定性 (Stability)**: 需求或技术变更频率低于使用频率
> 2. **通用性 (Generality)**: 适用于多个上下文
> 3. **封装性 (Encapsulation)**: 内部实现细节可被隐藏

### E.2 Cost-Benefit Threshold

> 复用的净收益存在阈值。设：
>
> - C_reuse: 复用成本（学习、适配、集成）
> - C_build: 自研成本
> - V_reuse: 复用带来的长期价值（维护、一致性、速度）
>
> 复用合理的条件是：C_reuse < C_build + V_reuse

## 3. 结构性公理 (Structural Axioms)

### S.1 Interface Substitution

> 两个组件可互相替换，当且仅当它们的**外部可观察行为**在给定约束下等价。

形式化：

```
C₁ ≃ C₂  ⟺  ∀ input ∈ Inputs, ∀ ctx ∈ Contexts:
    Obs(C₁(input, ctx)) = Obs(C₂(input, ctx))
```

### S.2 Compositionality

> 若组件 C₁ 和 C₂ 分别满足规约 S₁ 和 S₂，且它们通过兼容接口 I 组合，则组合体 C₁ ∘_I C₂ 满足 S₁ ∘ S₂。

### S.3 Dependency Transitivity of Trust

> 若 A 依赖 B，且 B 依赖 C，则 A 的信任边界必须扩展至 C。

形式化：

```
Trust(A) ⊆ Trust(B) ∩ Trust(C)
```

## 4. 过程性公理 (Process Axioms)

### P.1 Evolution Axiom

> 可复用资产的生命周期独立于任何单一使用它的系统。

### P.2 Feedback Axiom

> 复用资产的改进必须来源于使用者的反馈。没有反馈循环的复用资产将僵化。

### P.3 Governance Necessity

> 复用规模 N 与治理复杂度 G 的关系满足：G(N) = k × N × log(N)。当 G(N) 超过组织能力时，复用体系将崩溃。

---

## 5. 公理体系的批判与边界

> **公理的可证伪性**:
>
> - M.1 的反例：若存在一种复用完全不传递约束（如完全随机复用），则 M.1 失效。
> - M.2 的反例：若某类复用没有变性管理但仍成功（如完全相同的副本复用），则 M.2 需要限定范围。
> - M.3 的反例：若证明某两层完全可以互相替代，则 M.3 失效。

> **适用范围**: 本公理体系适用于**企业级软件系统**的架构复用。对于一次性脚本、探索性原型、极端定制化项目，公理的适用性减弱。
